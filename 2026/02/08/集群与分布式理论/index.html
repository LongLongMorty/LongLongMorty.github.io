集群与分布式理论

发表于2026-02-08|更新于2026-02-08|理论知识
|浏览量:

前言

最近想学习一些项目，发现都需要分布式知识，于是我找了一些资料学习。

集群（Cluster）和分布式（Distributed）就像是”人多力量大”的两种不同组织形式。虽然它们都涉及多台服务器，但出发点和解决问题的方式完全不同。

集群

集群就像是”多胞胎”。多台服务器物理上连接在一起，对外表现得像一台服务器。每台服务器运行的服务基本上是一模一样的。

  • 核心目的： 高可用（HA）和负载均衡（Load Balancing）
  • 工作方式： 当一个请求进来时，集群里的任何一台机器都能处理它。如果其中一台倒下了，另一台立刻顶上
  • 打个比方： 一家餐厅雇了 5 个厨师，每个厨师都会做全菜单的所有菜。客人来了，谁有空谁炒；一个厨师生病了，餐厅照样营业

集群应用示例

以一个支持千万用户同时在线的聊天应用为例：

在聊天场景下，集群意味着你部署了 100 台一模一样的”聊天连接服务器”。

  • 场景： 周五晚上，大家都上线聊天。如果只有一台服务器，它会因为连接数太多（CPU/内存爆满）直接宕机
  • 做法： 你弄一个”负载均衡器”（Load Balancer）放在最前面。用户 A 接入 1 号服务器，用户 B 接入 2 号服务器
  • 集群的威力： 即使 1 号服务器突然断电了，用户 A 只需要重新连接一下，负载均衡器就会把他分配到 2 号服务器。对用户来说，服务永远在线

分布式

分布式就像是”流水线”。它把一个庞大、复杂的业务拆分成多个小部分（微服务），分别部署在不同的服务器上。

  • 核心目的： 提高效率、解决单台机器性能上限（横向扩展）
  • 工作方式： 一个请求进来，可能需要 A 服务器算一下，B 服务器存一下，C 服务器发个通知，大家分工明确
  • 打个比方： 还是那家餐厅，现在规模大了。我们搞分工：有人专门切菜，有人专门炒菜，有人专门洗碗。每个人只负责一件事，但合起来完成一份订单

分布式应用示例

依旧以聊天软件为例，随着聊天软件变复杂，你发现一台服务器如果又要处理”发消息”，又要处理”文件传输”，还要处理”朋友圈视频渲染”，这台机器的性能会被榨干。

做法： 你把功能拆开，部署在不同的机器上：

  • 服务 A (存储服务)： 专门负责把聊天记录存进数据库
  • 服务 B (推送服务)： 专门负责给手机发送”叮咚”的消息提醒
  • 服务 C (图片处理)： 专门负责把你发的原图压缩成缩略图

分布式的威力： 当某天大家都在疯狂发朋友圈视频，只有”图片/视频处理服务”会很忙。你只需要给这个模块增加资源，而不影响别人发文字消息。

集群 vs 分布式对比

维度 集群 (Cluster) 分布式 (Distributed)
功能描述 相同的事，多个人做 不同的事，分摊着做
关系 纵向叠加（复制） 横向拆解（协作）
主要目标 可靠性（怕一台挂了） 性能（怕一台忙不过来）
耦合度 耦合度高，通常在局域网内 耦合度低，可以跨地域、跨网络

终极形态：分布式集群

在真实的聊天架构中，这两者是结合使用的：

  1. 分布式： 你把系统拆成了”消息服务”、”用户资料服务”、”支付服务”

  2. 集群：

    • 为了防止消息丢了，你搞了 10 台机器组成”消息服务集群”
    • 为了防止支付出错，你搞了 5 台机器组成”支付服务集群”

分布式不同模块（跨服务器）调用

当服务 A 和服务 B 运行在不同的物理机器上时，它们必须通过网络协议进行通信。

① 同步调用：RPC (远程过程调用)

这是最常见的方式，让调用远程服务就像调用本地函数一样简单。

  • 常用框架： gRPC, Dubbo, Spring Cloud OpenFeign
  • 逻辑： A 发起请求 → 挂起等待 → B 处理完返回结果 → A 继续执行
  • 特点： 响应快，但如果 B 挂了，A 也会跟着卡死

② 异步调用：消息队列 (MQ)

如果不要求立刻拿到结果，就用”发邮件”的方式。

  • 常用工具： RabbitMQ, Kafka, RocketMQ
  • 逻辑： A 把消息丢进队列 → A 直接去忙别的了 → B 有空了自己去队列取
  • 特点： 解耦、削峰填谷。比如聊天服务器里，发消息是实时的，但”存储聊天记录”可以通过 MQ 异步完成

同一模块不同进程调用

这种情况通常发生在一台性能很强的机器上（比如 64 核 CPU），为了充分利用资源，你开了多个相同的进程。这属于 IPC（进程间通信） 的范畴。

① 最简单：走网络回环 (Loopback)

即使在同一台机器上，也可以假装它们在不同机器。

  • 方式： 依然使用 HTTP 或 RPC，但 IP 地址写 127.0.0.1
  • 优点： 代码统一，不需要为”单机”和”分布式”写两套逻辑

② 最快速：共享内存 (Shared Memory)

  • 逻辑： 操作系统开辟一块内存区域，进程 A 往里写，进程 B 直接从里面读
  • 特点： 速度极快，不需要经过网卡，但需要处理”抢占”问题（加锁）

③ 最标准：Unix Domain Socket (UDS)

  • 逻辑： 在 Linux 系统里，这类似于一种特殊的”文件”
  • 特点： 比网络协议（TCP/IP）快得多，因为它省去了打包/拆包 IP 包头的性能损耗。数据库（如 MySQL）和 Web 服务器（如 Nginx）在本机通信时常用这个

进程通信示例

依旧以聊天软件为例：

  • 进程内通信： “长连接网关”为了抗住高并发，会在一台机器开 8 个进程。它们之间通过 Shared Memory 共享一份”在线用户名单”，效率最高
  • 跨模块通信： “网关模块”要把消息转发给”敏感词过滤模块”。如果要求极速过滤，就走 gRPC；如果只是要把消息”备份到数据库”，就丢进 Kafka (MQ) 慢慢存

实现序列化

无论是 RPC 还是消息队列，数据都不能直接把一个 Python 对象或 Java 对象扔过去，必须先把它**”拍扁”成二进制字节流。这个过程就是序列化**。

1. JSON：人人都爱，但它是”大胖子”

JSON 是目前最流行的格式，因为它可读性极强，直接打开文本就能看懂。

缺点： 冗余信息太多。比如 {"userId": 12345, "name": "Gemini"}

浪费在哪里：

  • 重复的 Key： 每次发消息都要带上 "userId" 这个字符串
  • 文本编码： 数字 12345 在内存里只占几个字节，变成字符串后占了 5 个字节
  • 解析慢： 机器读 JSON 需要逐个字符扫描、匹配括号

2. Protobuf：为了速度而生的”压缩大师”

Protobuf (Protocol Buffers) 是 Google 开发的一种二进制序列化协议。它不追求人能看懂，追求的是极致的体积和极致的速度。

它是怎么省空间的？

  • 不传 Key，只传编号： 在 Protobuf 的定义文件里，userId 被编号为 1。传输时，它只发一个编号 1，接收方对照表一看，就知道这是 userId
  • Varint 编码： 这是一种变长编码。如果数字很小（比如 5），它只用 1 个字节存，而不是固定用 4 个或 8 个
  • 二进制格式： 不需要处理引号、括号、逗号，机器读取速度极快

性能对比：

在一台普通的聊天服务器上：

  • 体积： Protobuf 通常只有 JSON 的 1/3 到 1/10
  • 速度： 序列化和反序列化的速度，Protobuf 通常比 JSON 快 5 到 10 倍

小比喻： JSON 就像是带包装的快递盒，里面有填充泡沫、说明书、精美外壳；Protobuf 就像是压缩真空包，抽干了所有空气，只剩下最核心的干货

3. 在聊天服务器里的实际应用

假设聊天系统每秒有 100 万条消息：

  • 如果用 JSON： 带宽成本可能是每月 10 万元，服务器 CPU 每天忙着解析字符串
  • 如果用 Protobuf： 带宽减半，服务器 CPU 占用率大幅下降，响应延迟（Latency）从 50ms 降到 10ms

这就是为什么像 gRPC 默认就使用 Protobuf 作为通信协议的原因。
