生产者-消费者模型

想象一个面包店的场景：

  • 生产者（线程 A） - 负责做面包，放到柜台上
  • 消费者（线程 B） - 负责从柜台拿面包吃
  • 柜台（共享资源） - 容量有限（比如只能放 5 个）

为什么需要这个模型？

如果老板（生产者）做太快，柜台满了，他得停下来睡觉；如果客人（消费者）发现柜台空了，他得坐着等，而不是一直盯着柜台看（那样太累了，浪费 CPU）。

核心工具：条件变量 (std::condition_variable)

在 C++ 并发编程中，std::condition_variable（条件变量）是实现线程间通信的核心工具。它允许一个线程在满足特定条件之前进入休眠状态，并在另一个线程修改了条件并发出通知时被唤醒。

条件变量主要有三类操作：等待（Wait）、通知（Notify） 和 锁定（Lock）。

1. 等待（Wait）

  • wait(lock) - 阻塞当前线程，直到被唤醒。调用时会自动释放锁，醒来时会自动重新获取锁
  • wait(lock, predicate) - 最推荐用法。predicate 是一个返回布尔值的函数（通常用 Lambda）。只有当条件为 false 时才等待，为 true 时直接通过。这可以有效防止”虚假唤醒”

2. 通知（Notify）

  • notify_one() - 唤醒一个正在等待该条件变量的线程
  • notify_all() - 唤醒所有正在等待的线程

问题分析

这个版本在粒度上还有很大改进空间。目前的写法是：

先拿锁 → 生产数据 → 睡觉 → 发通知 → 释放锁

相当于厨师占着灶台（锁），做完面包后还在灶台边眯了 500 毫秒，期间服务员（消费者）即使肚子饿得咕咕叫，也进不去厨房拿面包。

结果：程序变成了”串行”

  • 生产者睡觉时，消费者被锁堵住了
  • 消费者睡觉时，生产者被锁堵住了

核心准则：锁的粒度越小越好

在编写多线程代码时，遵循原则：“快进快出”

步骤 建议操作 理由
准备工作 锁外执行 耗时的计算、网络请求等不应占用锁
访问共享资源 锁内执行 必须保证 push/pop 过程是原子的
后续处理 锁外执行 日志打印、耗时模拟等放在锁外可以释放 CPU 给其他线程

关于 notify_one() 的位置

在改进版本中，我把 cv.notify_one() 挪到了锁的花括号后面。

两种方式对比

方式 执行流程 效率
放在锁内 被唤醒的线程（消费者）会立即尝试拿锁，但此时生产者还没释放锁，消费者醒了又得马上阻塞，白忙活一场 ❌ 较低
放在锁外 生产者先释放锁，再摇铃。消费者听到铃声直接就能拿到锁，效率更高 ✅ 更高

总结

生产者-消费者模型的核心要点：

  1. 使用条件变量 - 实现线程间的高效通信
  2. 最小化锁粒度 - 只在访问共享资源时加锁
  3. 锁外通知 - 将 notify_one() 放在锁外提高效率
  4. 避免虚假唤醒 - 使用带谓词的 wait() 函数
